# 維護方案評估：放棄舊網頁，改為「資料夾驅動」更新

> 目標：**不再維護舊 Word 網頁**。未來辦事員只要把檔案（PDF/Word/圖片…）放進對應資料夾，
> 按一下「更新程式」，程式自動偵測變動、更新網站。
> 撰寫：2026-07-02。前情見 `交接摘要.md`；同網域議題見 `搬移方案-同網域部署.md`。

---

## 一、這代表什麼轉變

| | 現況（維護模式 C） | 目標（資料夾驅動） |
|---|---|---|
| 資料來源 | 爬舊站 **HTML 網頁**（辦事員用 Word 改頁面） | 掃描**資料夾**（辦事員只放檔案，不碰網頁） |
| 檔案存放 | 舊 IIS 主機（`/01/`～`/20/`） | **待決**（本評估的核心分歧點，見下） |
| 辦事員操作 | 改 Word 頁 → 志工跑 sync → 志工 push | 放檔案 → 按更新程式 → **全自動上線** |
| 舊站角色 | 資料來源 + 檔案倉庫（不能關） | **可退役** |
| 標題/分類哪來 | 從 HTML 解析 | 從**資料夾位置＋檔名**解析（需命名規則） |

**關鍵洞察**：辦事員原本就要把檔案上傳到主機資料夾（`/01/`、`/02/`…），再手動改 Word 頁面加連結。
新方案等於**把「改網頁」這一步整個刪掉**——資料夾放哪、檔名叫什麼，就決定它出現在網站哪裡。
她已經熟悉這套資料夾代號與檔名格式（如 `115年7月.pdf`、`活動名(地點)--115.3.1.pdf`），學習成本低。

## 二、先決問題：檔案要住哪裡？（三個方案）

### 方案甲：全部搬進 GitHub repo（檔案＋網站同一處）⭐ 建議

檔案直接放在 repo 的 `files/01-安息日領會表/…` 等資料夾，由 GitHub Pages 一併服務。

- ✅ **單一來源**：全站 HTTPS，「不安全」標記、憑證問題、redirect.js **全部消失**
- ✅ 零伺服器、零費用；每個檔案都有版本紀錄（誤刪可救回）
- ✅ ntfy 通知、Pages 自動部署等既有機制直接沿用
- ⚠️ 額度限制：repo 建議 ≤1GB、單檔 ≤100MB（網頁拖放上傳 ≤25MB）、流量 100GB/月（軟限制，教會區域網站遠用不到）
- **可行性初判**：已盤點新站實際引用 **817 個檔案**，主力是 doc(277)/jpg(233)/pdf(192) 等小檔，
  可能大檔僅 mp4×1、rar×3、音檔×7、tif×3 → **總量極可能 <1GB**（Phase 0 實測確認；超標檔案個別處理，如音檔留存雲端/YouTube）

### 方案乙：檔案留在 IIS 當純檔案倉庫，外殼在 Pages

辦事員照舊把檔案丟 IIS 資料夾（不再改 Word 頁）；更新程式改成「列目錄」而非「爬 HTML」。

- ✅ 不受 GitHub 容量限制；檔案上傳流程完全不變
- ❌ 舊主機仍是單點依賴（本方案初衷就是想擺脫它）
- ❌ HTTP「不安全」問題依舊；需在主機開目錄列表或跑代理程式
- 適合：檔案量實測遠超 1GB 時的備援

### 方案丙：雲端硬碟（Google Drive / OneDrive）當檔案庫 — 不建議

分享連結不穩定（權限、改版、預覽頁體驗差）、無法用資料夾結構對映網址、需 API 金鑰維護。僅列出供完整性。

## 三、建議架構（方案甲）

```
repo（GitHub, main → Pages）
├── index.html / css/ / js/          ← 網站外殼（現有，不動）
├── files/                           ← ★ 新增：辦事員唯一要碰的地方
│   ├── 01-安息日領會表/115年7月.pdf
│   ├── 02-南區公文/兒童音樂營(高雄)--115.7.12.pdf
│   ├── 20-高屏小區公文/…
│   ├── 公告/本月公告.txt            ← 純文字公告（取代 Word 首頁中段）
│   └── …（沿用她熟悉的代號，後綴中文名）
├── data/*.json                      ← 由更新程式自動產生（人不編輯）
├── update.bat / update.py           ← ★ 一鍵更新程式
└── files-manifest.json              ← 變動偵測用清單（自動維護）
```

### 更新程式（update.bat）流程
1. **掃描** `files/` 全樹（路徑、大小、修改時間、雜湊）
2. **比對** `files-manifest.json` → 列出新增／修改／刪除；**無變動就直接結束**（「沒有變動，網站維持現狀」）
3. **重建** `data/links.json`（資料夾＝分類、檔名＝標題＋日期；沿用 `categorize.py` 既有的排序、
   安息日月曆、W/P 並排等邏輯）＋ `公告/本月公告.txt` → `announcements.json`
4. **驗證**：JSON 合法、數量門檻、單檔 >90MB 警告（同 sync.py 的保護哲學，失敗自動還原、不上線）
5. **git add / commit / push**：commit 訊息自動列出變動清單（例：「新增 2、修改 1：115年8月.pdf…」）
6. 完成訊息：「✓ 已更新，網站約 1–2 分鐘後生效」→ **你手機自動收到 ntfy 通知**（既有 workflow）

### 辦事員視角（全部操作）
> 把檔案放進正確資料夾 → 雙擊「更新網站」→ 看到綠色的「✓ 已更新」→ 完成。
> 公告文字：用記事本打開 `公告/本月公告.txt` 改字 → 一樣雙擊更新。

### 命名規則（需一頁說明書＋程式寬容處理）
- 已有慣例可直接沿用：`115年7月.pdf`（領會表）、`活動名稱(地點)--115.3.1.pdf`（公文）
- 不合規檔名**不會不見**：以檔名原樣當標題、進該資料夾對應分類，更新程式印提醒
- 建議避免：`#`、`%`、超長檔名（URL 相容性）

## 四、需要決策／確認的點

1. **容量實測**（Phase 0）：對 817 個檔案發 HEAD 請求加總大小，確認 <1GB、揪出 >25MB 的個別大檔 → 我可以直接跑
2. **辦事員電腦一次性安裝**：git + 推送憑證 + 桌面捷徑（志工/你到場或遠端設定一次；之後她不會再碰 git）
   - 備援選項：完全不裝軟體，改用 **GitHub 網頁拖放上傳**＋GitHub Action 在雲端自動重建 JSON（單檔 ≤25MB、需教她用網頁介面，介面是英文）
3. **重製內頁（doc-*.json 表格頁）怎麼辦**：建議**凍結為歷史彙編**（不再變動），新檔案一律進資料夾驅動的群組清單；若某頁（如講義彙編）未來仍要長大，再為它定義子資料夾規則（`08-講義彙編/事工類/109年/…` 可對映成表格分區）
4. **公告格式**：先用純文字檔（記事本可改）；日後可加一個本機 HTML 編輯器（交接摘要原本就預告的「日後再做編輯器」）
5. **舊站退役節奏**：檔案搬完、新站雙軌驗證 OK 後，舊站轉唯讀 → 首頁全頁導向新站 → 最終下線；同網域網址需求則接續《搬移方案-同網域部署.md》

## 六、風險與對策

| 風險 | 對策 |
|---|---|
| 總容量超過 1GB | Phase 0 先實測；超標的個別大檔留 IIS/雲端或 YouTube（音影檔本來就適合） |
| 單檔超過限制（push 100MB / 網頁 25MB） | 更新程式在驗證步驟直接擋下並提示 |
| 辦事員憑證過期/電腦更換 | 用不過期的 fine-grained PAT 或 SSH key；說明書寫入重設步驟 |
| 誤刪檔案 | git 歷史可完整還原；更新程式對「大量刪除」（如一次 >20 檔）要求再次確認 |
| 檔名不合規 | 寬容處理＋提醒，不會遺失內容 |
| push 失敗（斷網等） | 本機資料已就緒、無破壞；訊息提示「稍後再按一次即可」 |

## 七、Phase 0 盤點結果（2026-07-02 實測）

對新站引用的 817 個舊站檔案逐一 HEAD 請求（urllib + Mozilla UA，兩輪：8 併發 + 低併發補測）：

| 項目 | 數字 |
|---|---|
| 實際存在的檔案 | **684 檔，共 1.57 GB** |
| **確認 404 死連結** | **127 檔（約 15%！）**——舊站檔案早已被移走，現站上點了就是 404，遷移時順便清掉 |
| 暫時連不上（舊站不穩） | 6 檔（`/19/`、`/20/` 幾個 PDF，擇日重測即可） |

**1.57 GB 超過 Pages 1GB 額度——但幾乎全是 10 個大媒體檔造成的**：

- 講道錄音 mp3/m4a/wma ≈ **1,013 MB**（其中兩檔各 369 MB）
- 大合照 jpg、活動 mp4、超大 xls/pdf 等 >25MB 者共 10 檔 ≈ **1,124 MB**
- **排除這 10 檔後，其餘 674 檔僅約 480 MB** → 完全在額度內，且日常新增（公文 PDF 數百 KB）多年內無虞

**大檔安置**：這 10 檔放 **GitHub Releases**（同帳號、免費、單檔上限 2GB、永久網址、不佔 Pages 額度），
`links.json` 中這幾筆 URL 指向 Release 資產即可；日後罕見的新大檔（錄音/影片）同樣進 Releases 或 YouTube。

> 附帶收穫：127 個死連結是現站既有的體驗問題（點了 404），無論採不採本方案都值得清理。
> 明細存於盤點輸出 `size-audit.json` / `size-audit-retry.json`（scratchpad，需要可轉存 repo）。

## 八、結論

**方案甲確定可行**：正常檔案 480 MB 進 repo（額度綽綽有餘）＋ 10 個大媒體檔進 GitHub Releases。
附加紅利：HTTPS 問題整個消失、舊主機可退役、127 個死連結順勢清掉、每次變動有版本紀錄與手機通知。
主要成本是一次性的檔案遷移（Phase 1）與辦事員電腦設定（Phase 2），以及把「命名規則」變成正式約定。

**下一步**：業主點頭後即可啟動 Phase 1（鏡像抓檔進 repo、URL 改相對路徑、雙軌驗證）。
