# 搬移方案：讓新站落在相同網域（south.tjc.org.tw）

> 目標：辦事員希望新版手機網站的網址，不要停在 `huaihsin.github.io/south-tjc-mobile/`，
> 而是落在 `south.tjc.org.tw` 同一個網域之下。
> 撰寫：2026-06-26。配合 `交接摘要.md`、`README.md` 一起看。

---

## 一、為什麼不能「DNS 一指就好」——三個硬限制

1. **檔案資產綁在舊 IIS 主機**
   新站所有 PDF/Word/影音連結都指向 `south.tjc.org.tw/01/…/xx.pdf`，這些實體檔在舊的
   Windows **IIS** 主機上（big5、Word 匯出）。新站只是「外殼」，下載檔不在 repo 裡。

2. **`south.tjc.org.tw` 是 `tjc.org.tw` 的子網域**
   動它的 DNS 可能要透過**總會 IT**，屬跨單位協調，不一定能即時改。

3. **apex（根網域）不能用 CNAME 指向 GitHub Pages**
   DNS 規範下，`south.tjc.org.tw` 這個「裸網域」要指到 GitHub Pages 只能用 A record
   （GH 的固定 IP）。但那樣 apex 就**離開 IIS**，底下 `/01/`～`/20/` 的檔案全部 404，
   除非把所有檔案搬進 repo（檔案量大、又破壞辦事員直接改 Word 的流程）。

> 結論：**「精確就是 `south.tjc.org.tw` 這個網址」實務上只能在 IIS 主機上做（路線 B）。**
> 若可接受一個子網域（如 `www.south.tjc.org.tw`），則可用 GitHub Pages（路線 A）。

---

## 二、路線比較

| 項目 | 路線 A：子網域 → GitHub Pages | 路線 B：部署到 IIS（apex）⭐終局 |
|---|---|---|
| 對外網址 | `www.south.tjc.org.tw`（新增子網域） | `south.tjc.org.tw`（原網址不變） |
| 是否動 DNS | 要：加一筆 CNAME（需總會 IT 協調） | **不用**：檔案放上既有主機即可 |
| HTTPS | GitHub 自動免費憑證 ✅ | 自行在 IIS 安裝憑證（Let's Encrypt / win-acme） |
| 檔案連結安全性 | 仍連 http apex → 開檔該分頁標「不安全」（同現況） | **單一來源、無混合內容**，全站同協定 |
| 部署方式 | 維持現有 `git push` 自動部署 ✅ | 需建立「把靜態外殼送上 IIS」的機制 |
| 回滾 | GH 設定移除自訂網域即還原 | 保留舊 `index.htm`，移除預設文件即還原 |
| 適合情境 | 想快上線、要免費 HTTPS、DNS 能協調 | 要真正同網址、單一來源、有主機存取權 |

**建議**：拿得到 IIS 主機存取權 → 走 **B**（最乾淨的終局）；短期動不了主機 → 先走 **A** 過渡，日後收斂到 B。

## 三、路線 A：子網域 → GitHub Pages（快、免費 HTTPS）

### 步驟
1. **選子網域**，建議 `www.south.tjc.org.tw`（或 `m.` / `new.`）。
2. **DNS**：在管理 `south.tjc.org.tw` 的 DNS 區域新增一筆
   `CNAME  www  →  huaihsin.github.io`（需該網域 DNS 管理者／總會 IT 操作）。
3. **repo 綁定自訂網域**：
   - 在 repo 根目錄新增檔案 `CNAME`（無副檔名），內容單一行：`www.south.tjc.org.tw`。
   - GitHub → repo Settings → Pages → Custom domain 填入該網址 → 等 DNS 檢查通過。
   - 勾選 **Enforce HTTPS**（GitHub 會自動申請 Let's Encrypt 憑證，約數分鐘～數小時）。
4. **更新 redirect.js / 舊站入口**：把舊站彈窗導向改成新網址（只改 `redirect.js` 一個檔）。
5. （可選）讓 apex 轉址到子網域：在 IIS 加一條 URL Rewrite，把 `south.tjc.org.tw` 301 → `www.south.tjc.org.tw`。

### 仍存在的小問題
- 新站是 HTTPS，但檔案連結仍是 `http://south.tjc.org.tw/…`。點下載是**最上層導覽**（非 mixed-content），檔案能開，但那個分頁會標「不安全」——與現況相同。要徹底消除，仍需替 apex 裝 HTTPS（見路線 B 的 HTTPS 段）。

### 回滾
- GH Settings 移除 Custom domain ＋ 刪 DNS CNAME → 立刻回到 `*.github.io` 網址。

## 四、路線 B：部署到 IIS、落在 apex（同網址、單一來源）⭐

### 前置
- 需要 IIS 主機的**檔案系統存取**與 **IIS 管理員**權限（或請主機管理者配合）。
- 保留舊 `index.htm`（Word 首頁）：它仍是 `home_bulletin.py` 解析「本月公告」的**資料來源**，也是檔案託管根目錄，**不要刪**。

### 步驟
1. **部署靜態外殼到網站根目錄**
   把 repo 的 `index.html`、`css/`、`js/`、`data/`、`assets/`、`redirect.js`
   複製到 IIS 站台根目錄（與 `/01/ /02/ …` 同一層）。

2. **設定預設文件順序**
   IIS Manager → 該站台 → Default Document → 把 `index.html` 排在 `index.htm` **之前**
   （根目錄就會先送新站；舊 `index.htm` 仍可用 `/index.htm` 直接存取）。

3. **補 MIME types（重要）**
   IIS 預設可能不送 `.json` / `.webmanifest` 等，導致 `fetch('data/links.json')` 失敗。
   在 IIS 加上：
   - `.json` → `application/json`
   - `.webmanifest` → `application/manifest+json`
   - `.svg` → `image/svg+xml`（通常已內建）
   - `.woff2` → `font/woff2`（若有用到自訂字型）

4. **切換連結 base 為同來源**
   既然同一台主機，把連結改成同來源即可消除混合內容：
   - `data/links.json` 的 `meta.base`、以及 `crawl/categorize.py` 的 `BASE`
     改為 **根相對** `"/"`（即 `/01/xx.pdf`），或維持絕對網址但改成 `https://south.tjc.org.tw/`。
   - 建議用根相對 `/`，跟著站台協定走，最不會出錯。改完重跑 `sync.py` 重新產生資料。

5. **修好 HTTPS（解決舊站 443 被重置）**
   - 最省成本：用 **win-acme**（Let's Encrypt 的 Windows 客戶端）申請免費憑證，自動續期。
     需 80 埠對外可達做 HTTP-01 驗證（目前 HTTP 正常 → 可行）。
   - 或安裝總會/單位既有的憑證，於 IIS 綁定 443。
   - 綁好後加 HTTP→HTTPS 轉址（IIS URL Rewrite）與 HSTS。
   - 全站變 HTTPS 後，新站外殼與檔案連結同協定 → **不再有「不安全」標記**。

6. **退役 redirect.js 彈窗**
   既然網址沒變、且根目錄已是新站，原本「導向新版」的彈窗不再需要：
   移除舊 `index.htm` 末端那行 `<script src=…redirect.js>`（或清空 `redirect.js`）。

### 日後部署機制（擇一）
- **伺服器端 git pull**：在 webroot clone 本 repo，更新外殼時於主機 `git pull`。
- **GitHub Action FTP/SFTP 部署**：push 到 main 時，用 action 把靜態外殼上傳到 IIS。
  （可與現有的 ntfy 通知 workflow 並存。）
- 辦事員改 Word 的 `/01/…` 檔案流程**完全不變**；`data/*.json` 由 `sync.py` 重新產生後一起部署。

### 回滾
- 保留舊 `index.htm`：把它在 Default Document 排回最前（或移除 `index.html`）
  → 根目錄立即回到舊站，新站檔案留著不影響。

## 五、決策點（請辦事處/維護者確認）

1. **要的是 apex 還是可接受子網域？** apex → 路線 B；可接受 `www.` → 路線 A 也行。
2. **有沒有 IIS 主機的存取權／能否請主機管理者配合？** 有 → B 可行。
3. **`south.tjc.org.tw` 的 DNS 由誰管？** 走 A 要改 CNAME，需這個人。
4. **誰負責裝/續 HTTPS 憑證？** B 的單一來源安全體驗倚賴這步（也是現有待辦第 1 項）。

---

## 六、建議路徑

1. **短期（數天內可上線）**：路線 A —— 子網域 + GitHub Pages，拿到免費 HTTPS、維持自動部署。
2. **終局（最乾淨）**：路線 B —— 新站落在 `south.tjc.org.tw` 本身、單一來源、修好 HTTPS、退役 redirect.js。
3. 兩者非互斥：A 上線後，待主機與憑證就緒，再平滑收斂到 B（apex 301 轉址或直接接管根目錄）。
